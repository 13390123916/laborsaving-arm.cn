## 三、P0 级（收录阻断 · 上线前必须解决）

### P0-1 站长验证与统计 ID 全部为占位符 → 收录无法触发

- **现象**（实跑 `GET /api/site-config/info/`）：
  ```
  baidu_tongji : 0000000000000000
  baidu_verify : code-labor-saving-baidu-verify
  qihu_verify  : code-360-qihoo-verify
  sogou_verify : code-sogou-site-verify
  google_verify: google-search-console-verify-code
  gtm_id       : GTM-0000000
  google_ga_id : G-0000000000
  ```
- **复现**：前端 `seo.js` 会把这些值注入 `<meta>` 与统计脚本；实测渲染后 `baidu-verify / qihu-verify / sogou-verify / google-verify` 全部为上述占位串。
- **影响（SEO/GEO）**：
  - 百度/360/搜狗/Google **站长验证 meta 内容无效** → 各站长平台**验证失败**，无法提交 sitemap、无法开启主动推送 → **搜索引擎不收录或延迟收录**。
  - 百度统计/GA/GTM 用占位 ID → 统计脚本虽加载（百度 `hm.js` 对假 ID 返回 200），但**数据指向不存在的站点，统计为零**。
  - 这是整个"收录优化目标"的**第一阻断点**：代码链路正确，但缺真实凭证。
- **修复建议**：
  1. 在百度/360/搜狗/Google 站长平台获取**真实验证 code** 与**统计 ID**；
  2. 经 Django Admin（见 P0-2）填入 `SiteConfig` 对应字段；
  3. 在 `docs/SEO配置说明.md` 中固化"上线前必须替换 7 个占位值"的检查清单。

### P0-2 无超级管理员账号 → 后台管理闭环断裂，且无法填真实验证码

- **现象**：`python3 manage.py shell` 直查 `User` 表 → **用户总数 0、超管 0**。`api/init_data.py` 中**不含** `create_superuser`。
- **复现**：`git` 干净；按 README 执行 `migrate` + `init_data.py` 后，`/admin` 无任何可登录账号。
- **影响**：
  - 项目核心承诺"所有页面数据后台可配置、可更新"**不成立** → FAQ/资讯/验证码/统计 ID 均无法在后台维护。
  - 与 P0-1 **强关联**：没有超管 → 无法在后台填入真实验证 code → P0-1 无法闭环。
  - "后台管理"交付标准未达成。
- **修复建议**：在 `init_data.py` 末尾增加幂等超管创建（或独立 `management/command`）：
  ```python
  if not U.objects.filter(username='admin').exists():
      U.objects.create_superuser('admin', 'admin@laborsaving-arm.cn', '请改为强密码')
  ```
  并写入 `docs/部署文档.md` 的"首次部署"步骤。`SECRET_KEY`/密码后续改环境变量（见 P2-9）。

### P0-3 `SERVE_SPA` 部署模式未按路由提供预渲染快照 → 子路由全返回首页 HTML

- **现象**（实跑 `SERVE_SPA=True` 模式）：
  - `dist/` 内 SSG 生成了 `dist/about/index.html`、`dist/faq/index.html`、`dist/news/1/index.html` 等**按路由快照**；
  - 但 `curl /about` 返回 11491 字节、**含首页标题"气动助力机械臂厂家"与首页 FAQ 预览文本**、**不含关于页实体文本（资质实力/成立时间）**。
  - 即：所有子路由都被兜底逻辑返回 `dist/index.html`（= 预渲染的**首页**），而非各自快照。
- **根因**：`config/urls.py` 的 `spa_fallback` 永远 `open(dist/index.html)`，未优先读取 `dist/<route>/index.html`。
- **影响（SEO/GEO，严重）**：
  - 爬虫/AI 引擎访问 `/about`、`/faq`、`/news/1` 都拿到**首页的 TDK 与正文**；
  - 逐页独立标题/关键词/描述/JSON-LD **对不执行 JS 的爬虫全部失效**；
  - 预渲染（SSG）投入的 11 条快照**在 SERVE_SPA 部署下形同虚设**；
  - GPTBot/CCBot 等 AI 爬虫抓取到的页面内容错位 → GEO 引用质量崩塌。
- **修复建议**（改 `config/urls.py` 的 `spa_fallback`）：
  ```python
  import os
  def spa_fallback(request, **kwargs):
      path = request.path.lstrip('/')
      # 1) 优先返回按路由预渲染快照
      cand = os.path.join(str(dist_dir), path, 'index.html')
      if os.path.exists(cand):
          return HttpResponse(open(cand, encoding='utf-8').read())
      # 2) 兜底返回 SPA 入口（含预渲染首页或空壳）
      if index_path and os.path.exists(index_path):
          return HttpResponse(open(index_path, encoding='utf-8').read())
      return HttpResponse('SPA not built.', status=503)
  ```
  部署前务必 `pnpm run build:prerender`（已验证 11/11 成功），再开 `SERVE_SPA=True`。

## 四、P1 级（严重影响收录质量/一致性）

### P1-4 360 自动收录脚本 URL 返回 404（死链）

- **现象**（无头渲染失败请求 + 实跑 curl 双重确认）：
  ```
  HTTP 404 https://s.ssl.qhres2.com/ssl/ab77b6ea7ac43b89.js
  ```
  外网可达（百度/百度推送均 200），唯独该 360 `sozz` 加载器 **404**。
- **根因**：`frontend/src/plugins/seo.js` 硬编码了该 `qhres2.com` 脚本地址，当前已从源站失效。
- **影响**：360 搜索引擎**主动收录（自动推送）失效** → 360 收录量与时效下降。
- **修复建议**：到 **360 站长平台**重新获取最新 `sozz` 脚本地址并替换；建议把该 URL 也抽到 `SiteConfig` 配置项，避免再次硬编码死链。

### P1-5 `/about` 路由 keywords 含"山东机械臂厂"，与全站"辽宁沈阳"矛盾

- **现象**：`router/index.js` 中 `/about` 的 `keywords: '...LABOR-SAVING,企业资质,山东机械臂厂'`。
- **影响**：企业真实地址为**辽宁省沈阳市浑南区文溯街19-1号**（DB 与 `index.html` 地理 meta 均一致），但关于页关键词写"山东" → **地理信号自相矛盾**，干扰百度本地搜索/地图收录与用户信任。
- **修复建议**：改为"辽宁机械臂厂/沈阳机械臂厂家"，与 DB `address_region/address_locality` 对齐；建议关键词由 `SiteConfig` 驱动而非写死。

### P1-6 结构化数据地址 fallback 写死"青岛市/山东省"

- **现象**：`api/seo_urls.py` 与 `frontend/src/composables/useJsonLd.js` 的 `addressLocality/addressRegion` 默认值为 `'青岛市'/'山东省'`。
- **影响**：当前 DB 已填沈阳，故实际输出正确；但**一旦配置字段为空，JSON-LD 会向外宣称"青岛/山东"**，与全站沈阳定位冲突，造成 GEO 结构化数据错误（AI 引擎可能据此误判属地）。
- **修复建议**：fallback 改为沈阳/辽宁，或直接 `required` 校验，杜绝矛盾默认值。

### P1-7 关于页（企业实体卡）未注入 LocalBusiness JSON-LD

- **现象**：无头渲染实测 `/`(LocalBusiness)、`/faq`(FAQPage)、`/news/1`(Article) 均有 JSON-LD；**`/about` 与 `/contact` 的 JSON-LD 为空**。
- **影响**：关于页是**企业实体背书与本地 SEO/GEO 最关键页面**，却缺少 `LocalBusiness/Organization` 结构化数据 → 搜索引擎与企业知识图谱（Google Knowledge Panel）、AI 实体引用**拿不到结构化企业信息**。
- **修复建议**：在 `About.vue` 挂载时调用 `useJsonLd('organization').inject(buildOrganizationSchema(siteConfig))`，与首页/后端 `/schema/organization.json` 对齐。

## 九、功能缺失专项审查（2026-07-19 补充）

| 密钥权限分级 | ❌ 缺失 | 无读写权限控制、接口级别授权 |

**影响**：所有 API 接口（含表单数据查询）对外完全开放，无法对接外部系统（ERP/CRM）或开放部分 API 给合作伙伴。

### 9.5 登录日志 → ❌ 完全缺失

| 子功能 | 状态 | 说明 |
|--------|------|------|
| 登录日志模型 | ❌ 缺失 | 无 `LoginLog`/`AuditLog` 模型 |
| IP 记录 | ❌ 缺失 | 无登录 IP 追踪 |
| 操作记录 | ❌ 缺失 | 无敏感操作审计（文章修改、配置变更、数据删除） |
| 日志查询后台 | ❌ 缺失 | Admin 无日志查看入口 |

> 注：Django 内置 `django_admin_log` 表会记录 Admin 后台的增删改操作，但未被项目代码显式配置或暴露，且不记录登录事件本身。

**影响**：无法追溯"谁在什么时候做了什么"，安全审计缺失；发生误操作或数据泄露无法定位。

### 9.6 域名与 SSL → ❌ 完全缺失

| 子功能 | 状态 | 说明 |
|--------|------|------|
| 域名绑定配置 | ❌ 缺失 | 无域名管理模型，`ALLOWED_HOSTS=['*']` 硬编码 |
| CDN 加速 | ❌ 缺失 | 无 CDN 配置或集成 |
| SSL 证书管理 | ❌ 缺失 | 无证书配置功能（`Certificate` 模型是企业资质证书，非 SSL） |
| 全站备份 | ⚠️ 基础 | `scripts/backup.sh` 提供 SQLite 数据库备份/恢复/30 天保留，但不含代码/媒体文件备份 |

**影响**：域名、CDN、SSL 属于运维层面，通常不在应用代码内管理；但当前 `ALLOWED_HOSTS=['*']` + `DEBUG=True` 的组合存在安全隐患。备份仅覆盖数据库，不含上传文件（因为目前也没有文件上传功能）。

### 9.7 安全设置 → ❌ 严重不足

| 子功能 | 状态 | 说明 |
|--------|------|------|
| IP 白名单 | ❌ 缺失 | 无 IP 限制中间件或配置 |
| 防刷/限流 | ❌ 缺失 | DRF 无 `DEFAULT_THROTTLE_CLASSES`，表单接口可被无限刷 |
| 暴力登录拦截 | ❌ 缺失 | 无 `django-axes` 等登录保护库，Admin 登录无失败次数限制 |
| 全站备份 | ⚠️ 基础 | 仅 SQLite 文件备份脚本 |
| HTTPS 强制 | ❌ 缺失 | 无 `SECURE_SSL_REDIRECT`、`SECURE_HSTS_SECONDS` 等安全头配置 |
| CSRF 保护 | ✅ 已有 | Django 内置 CSRF 中间件 |
| XSS 保护 | ✅ 已有 | Django 模板自动转义 + Vue 默认转义 |

**影响**：表单接口可被恶意刷量；Admin 后台登录无爆破保护；生产环境缺少 HTTPS 安全头配置。
